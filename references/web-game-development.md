# 已批准 GDD 的网页游戏开发合同

本合同只把已明确批准的 GDD 变成可运行、可验证的儿童友好网页游戏。

## 进入门禁与权威输入

开始任何实现、依赖安装或游戏文件创建前，必须同时确认：

- 账本明确记录 `gdd_approved: true`；
- 输入是点名的已批准 `gdd_version`，页眉状态为 `已批准`；
- 开发计划逐项引用该版本的最小可玩范围和验收标准。

任一条件不满足就停止开发，返回 GDD 审阅。实现中发现规则不可执行、范围冲突或需要改变体验时，也暂停相关功能，回到 GDD 修改、递增版本和明确批准循环；不得静默改变已批准的体验。

## Web-only 交付与默认技术栈

最终产品必须是静态、离线可运行的 Web 游戏，唯一入口为 `game/index.html`。不得交付桌面安装包、原生移动应用，或 Unity、Unreal、Godot 工程。玩家不需要安装 Node.js、Python、npm、数据库或专用启动器。

默认使用原生 HTML5、CSS 和 JavaScript：

- 2D 使用 DOM、Canvas 2D 或二者组合；
- 所有脚本、图片、字体和声音随 `game/` 本地交付，不依赖 CDN；
- 不使用运行时 `import`、`fetch` 和运行时网络请求；
- `game/index.html` 必须能从文件管理器直接双击离线运行；
- 本地 HTTP 服务只用于开发或测试，不得成为玩家运行前提。

## 按需轻量 3D

如果学习者的创意或已批准 GDD 确实需要空间、深度或立体探索，可以实现 3D，但仍必须是低负载 Web 游戏：

- 使用浏览器 WebGL；可使用一个随项目本地交付的 Three.js 类轻量库；
- Three.js 类库使用可由普通 `<script>` 引入的本地浏览器单文件包，或在开发期预先打包为单一普通脚本；玩家端不运行打包器；
- 优先使用程序生成几何体；不在运行时加载 GLTF、远程纹理或其他模型资源，必须使用的模型或纹理应预打包或内嵌，并实际验证 `file://`；
- 不使用 WebGPU、WASM、GPU 计算或桌面级 3D 引擎；
- 限制模型面数、材质、动态光源和同时可见物体，使用小尺寸压缩纹理；
- 默认关闭实时阴影、复杂物理、全屏后处理、高开销粒子和大量透明物体；
- 提供降低画质或减少效果的路径，并在集成显卡设备上验证核心流程；
- 如果 DOM 或 Canvas 2D 足以表达玩法，不得只为视觉炫技升级到 3D。

不使用 Phaser、React、Vue、npm 构建链或打包器作为玩家运行方案。已批准 GDD 已明确需要 3D、WebGL 和一个本地 Three.js 类轻量库时，可直接按该版本实施，无需重新审批。只有新增依赖改变体验、范围或资源预算时，才返回 GDD 审阅并重新批准。

## 兼容、资源与数据边界

- Windows 10/11：当前稳定版 Chrome、Edge 或 Firefox；
- macOS：当前稳定版 Safari、Chrome 或 Firefox；
- 不要求独立显卡，2D 与 3D 都必须在普通集成显卡档电脑可玩；
- 桌面 1280×720 与手机 390×844 视口都能完成核心流程；
- 首版全部资源不超过 10 MB；
- 普通设备目标 60 FPS，低性能设备核心流程保持可玩的 30 FPS；
- 首个版本不使用后端、服务器、数据库、登录、联网多人、支付、广告、公开聊天、上传、追踪、云函数或外部 API。

只有新的 GDD 版本明确写出必要性、安全边界和验收标准，并再次获批，才能改变数据边界；离线启动、儿童隐私和多数 Windows/macOS 电脑兼容要求不得取消。

## 儿童内容安全实现边界

实现必须保留获批 GDD 的儿童安全边界，不得加入或细化可操作的现实危险、自残、仇恨式针对、性内容、赌博或真钱随机机制、战利品箱操纵、暗黑模式或剥削性变现。相关玩法只能采用虚构、非血腥、非性化、不歧视且不使用真钱的安全版本；任何会突破这些边界的实现请求都停止开发并返回安全的 GDD 审阅。

## 测试驱动开发循环

每一项 GDD 验收标准和每一个缺陷都遵循同一循环：

1. 先写对应的自动化验收测试；无法自动化的视觉或触控行为，先在 `game/PLAYTEST.md` 写出可重复的失败验证步骤。
2. 运行测试或验证步骤，看到由缺失行为导致的预期失败。
3. 只实现让该项通过的最小行为。
4. 重新运行并确认通过，同时确认已有测试仍为绿色。
5. 只在绿色状态下重构，再次运行验证。

没有先观察到预期失败，不得把测试记作开发证据。实现冲突要回到 GDD 审批，不能通过修改测试来掩盖。

## 默认交付结构

```text
game/
├── index.html
├── styles.css
├── src/
├── tests/
├── assets/
├── README.md
└── PLAYTEST.md
```

最终交付物必须包含：

- `game/index.html` 以及 `game/` 内可离线运行的源代码和本地资源；
- `game/README.md`，把“直接双击 `index.html`”作为第一运行方式，并说明测试命令、浏览器要求和键盘/触控操作；
- `game/tests/` 内与每条 GDD 验收标准对应的自动化测试；
- `game/PLAYTEST.md`，记录人工与浏览器验证证据；
- 获批的最终 GDD、`gdd_version` 和批准状态。

## 无障碍与响应式要求

- 桌面与手机布局都能完成核心流程，内容不被遮挡或截断；
- 核心玩法可用键盘完成，焦点顺序合理且焦点样式清晰可见；
- 手机提供与键盘操作等价的触控控制，触控目标大小和间距适合儿童；
- 文字、控件与背景有清晰对比度；
- 不使用只能通过悬停触发的操作；
- 有明显移动、闪烁或持续动画时，响应减少动态效果的系统偏好，并提供适当的动效减弱。

## 运行时合同

游戏必须可开始；支持暂停或重新开始；能到达胜利与失败；对关键动作、进展、错误、胜利和失败给出清楚反馈。页面加载和完整核心流程中不得出现未处理异常或控制台错误。

## 浏览器验证

在交付前完成桌面视口和常见手机视口的完整试玩，并记录尺寸、浏览器和操作系统：

1. 断开网络，从直接双击 `game/index.html` 开始，走通开始、核心循环、暂停或重新开始、胜利和失败。
2. 在桌面验证键盘控制，在手机视口验证触控等价控制。
3. 检查布局、可见焦点、对比度、无悬停依赖和适用的动效减弱。
4. 打开并检查控制台，确认没有未处理异常或控制台错误。
5. 检查网络面板，确认无 CDN、外部 API 或其他运行时网络请求；检查相对资源在 Windows 风格和 macOS 风格路径条件下可加载。
6. 记录资源总量和性能；3D 还要记录集成显卡档条件、画质降低路径及 30 FPS 可玩性。
7. 将获批 GDD 的每一条验收标准逐项与自动化或人工证据对照；没有证据的条目不得标记通过。

## 每次完成后的自动预览

每次完成初版或一次迭代并且测试为绿色后，必须立即启动本地预览，再报告完成：

```bash
python3 <本 Skill 路径>/scripts/preview_game.py --project game
```

该命令先检测端口冲突：优先使用 `5173`，被占用时依次尝试 `3000`、`8080`、`8000`，最后让系统分配空闲端口；随后自动打开默认浏览器。必须把实际预览 URL、任何跳过的端口及浏览器打开结果记录到 `game/PLAYTEST.md`。如果浏览器未能自动打开，立即报告可点击的本地 URL 并自行在可用浏览器中打开它，再继续试玩或交付。

本地预览只用于开发、测试和第一时间打开项目；它不得替代离线双击 `game/index.html` 的交付验证，也不得成为玩家运行游戏的前提。

## `game/PLAYTEST.md` 记录格式

`game/PLAYTEST.md` 必须包含：

- 已批准 GDD 版本；
- 测试环境，包括浏览器、操作系统和日期；
- 运行与测试命令；
- 每次完成初版或迭代后的预览 URL、端口冲突检测结果和浏览器打开结果；
- 自动化测试结果；
- 人工试玩步骤与实际结果；
- 桌面和手机视口尺寸及各自结果；
- 直接双击离线启动、Windows 10/11 与 macOS 浏览器兼容结果；
- CDN、外部 API 和运行时网络请求检查结果；
- 首版资源总量与性能结果；3D 项目还要记录集成显卡档测试和降质路径；
- 键盘与触控等价控制结果；
- 控制台检查结果；
- GDD 验收标准到证据、结果的验收矩阵；
- 已知限制及其对体验的影响。

失败项必须记录可复现步骤，回到测试驱动循环修复并重新验证。

## 资源规则

只使用原创、程序生成、许可明确或已获授权的资源，并在 `game/README.md` 记录来源与许可。不得复制在世艺术家的独特风格，不得使用未经许可的商业图片、字体、音乐、音效或其他素材。无法确认许可时，改用原创或程序生成的占位资源。
