# 青少年网页游戏创作 Skill 设计规格

## 1. 项目目标

创建一个名为 `youth-game-builder` 的 Codex Skill。当用户说“我想创作一个游戏”或表达相同意图时，Skill 面向小学、初中阶段的学生启动一段循序渐进的游戏设计对话：先完成不少于 20 轮、与此前回答持续关联的有效问答，再生成完整的 GDD（Game Design Document，游戏设计文档），经用户明确批准后，才开始开发并验证可运行的网页游戏。

本项目最终交付到 GitHub 仓库 `Nagi1998/game-builder-skill`。

## 2. 成功标准

完成后的 Skill 必须满足以下可验证行为：

1. 精确触发语“我想创作一个游戏”能触发 Skill；“帮我设计游戏”“我有一个游戏想法”“想做网页游戏”等同义意图也能触发。
2. 对话使用适合小学、初中生理解的中文，一次只提出一个主要问题。
3. 每个后续问题都基于已收集信息，引用、概括或对照至少一项此前回答，并推动细化、取舍、冲突消解或可实现性判断。
4. 只有包含有效设计信息的学生回答才计为一轮；助手消息、确认词、无法理解的回答和重复内容不计入有效轮数。
5. 在完成至少 20 轮有效回答前，不生成最终 GDD，也不开始游戏开发。
6. 达到 20 轮后仍要检查 GDD 信息覆盖；存在关键缺口或矛盾时继续追问，因此 20 轮是下限而不是自动结束点。
7. 最终 GDD 覆盖愿景、目标玩家、核心玩法循环、规则、操作、角色、世界、关卡、资源与奖励、难度、失败与胜利、界面、美术、声音、无障碍、安全、技术范围和验收标准。
8. GDD 输出后进入修改与审批循环；只有用户明确表达“批准 GDD”“按这个开发”等同意意图后才能写游戏代码。
9. 开发阶段交付可运行、响应式、键盘可操作、适合儿童使用的网页游戏，同时包含运行说明与试玩验证记录。
10. Skill 本身通过结构检查、行为合同检查、基线对照评测和触发测试，并在推送后核验远端内容。

## 3. 目标用户与交流原则

### 3.1 目标用户

- 小学生和初中生是主要创作者。
- 学生不需要掌握游戏设计、编程或 GDD 术语。
- Skill 负责把学生的自然表达转化为专业设计决策，但不能替学生偷偷做关键创意选择。

### 3.2 表达方式

- 使用短句、具体例子和生活化比喻解释专业概念。
- 每条消息只问一个主要问题；必要时可列出 2–4 个选项，并始终允许学生提出自己的答案。
- 先肯定回答中有价值的部分，再指出需要继续思考的设计问题；不使用考试式评价或贬低表达。
- 每完成约 5 轮有效回答，单独给出一段简短的“目前我们已经决定了什么”，并把“哪里需要纠正”作为这一条消息的唯一问题；确认完成后，下一条消息再继续设计问题。单纯确认小结不计入有效轮数，只有学生补充或修改设计决定时才计数。
- 学生说“不知道”时，给出对比明显的例子或缩小选择范围，再继续询问，不直接代答。

## 4. 总体架构

采用“一个入口 Skill + 按需加载参考资料 + 确定性校验脚本 + 评测数据”的模块化结构：

```text
game-builder-skill/
├── SKILL.md
├── README.md
├── references/
│   ├── interview-protocol.md
│   ├── gdd-template.md
│   └── web-game-development.md
├── scripts/
│   └── validate_skill.py
├── tests/
│   └── test_skill_contract.py
├── evals/
│   └── evals.json
└── docs/
    ├── implementation-plan.md
    └── superpowers/specs/2026-07-16-youth-game-builder-design.md
```

各模块职责：

- `SKILL.md`：触发后的总流程、状态机、硬门禁、资源读取时机和快速参考。
- `interview-protocol.md`：有效轮次定义、主题覆盖表、关联追问算法、年龄适配和异常回答处理。
- `gdd-template.md`：覆盖检查清单、最终 GDD 固定结构和审批规则。
- `web-game-development.md`：技术选型、项目结构、开发步骤、质量标准、浏览器验证和交付物。
- `validate_skill.py`：无第三方依赖地检查目录、元数据、关键合同文字、问答下限、GDD 与开发门禁。
- `test_skill_contract.py`：先于校验脚本实现的自动化合同测试，用于证明校验逻辑能发现缺失项。
- `evals.json`：真实用户提示、预期行为和可客观检查的评测断言。

`SKILL.md` 保持为轻量流程入口，较长规则放入 `references/`，避免每次触发都加载全部细节。

## 5. 会话状态模型

Skill 在对话中维护一份内部“设计账本”。账本不是让学生填写的表单，而是助手根据每次有效回答更新的结构化记录。

### 5.1 状态字段

```text
phase: discovery | interview | gdd-review | approved | development | verification | complete
valid_rounds: 非负整数
age_band: primary | middle | unknown
decisions: 已确认的设计决定
open_questions: 尚未解决的问题
conflicts: 回答之间的冲突
coverage: GDD 各主题的 complete | partial | missing
gdd_version: 当前 GDD 版本号
gdd_approved: true | false
development_status: not-started | building | verifying | done
```

### 5.2 状态转换

1. `discovery`：欢迎学生并询问年级段或希望使用的解释难度，不收集真实身份信息。
2. `interview`：进行关联式设计问答并累计有效轮数。
3. 当 `valid_rounds >= 20` 且 GDD 必需主题全部覆盖、冲突已解决时，进入 `gdd-review` 并生成 GDD。
4. 用户要求修改 GDD 时，保持 `gdd-review`，更新文档版本后再次请求批准。
5. 只有明确批准当前版本时，设置 `gdd_approved: true` 并进入 `approved`。
6. `approved` 后读取网页开发规范，创建实现计划并进入 `development`。
7. 游戏功能完成后进入 `verification`；通过自动化和浏览器试玩验证后进入 `complete`。

任何状态下都不得反向推断批准。沉默、“差不多”、单纯继续讨论或只确认局部章节，都不等于批准完整 GDD。

## 6. 不少于 20 轮的关联问答协议

### 6.1 主题覆盖

问答不是固定念完 20 道题。Skill 根据学生回答在以下六个主题间动态选择下一项：

1. 创意愿景：幻想、情绪、主题、参考体验、独特之处。
2. 玩家与目标：目标玩家、玩家扮演谁、短期目标、最终目标。
3. 核心玩法：主要动作、核心循环、规则、反馈、胜利和失败。
4. 世界与内容：故事背景、角色、地点、关卡、敌人、谜题或任务。
5. 系统与体验：资源、奖励、成长、难度、界面、操作、美术和声音。
6. 实现边界：单局时长、设备、单人或多人、最小可玩版本、可测试标准。

每个主题都必须达到 `complete`。主题不按固定轮数平均分配；学生的具体回答可能一次覆盖多项，含糊回答则需要多轮追问。

### 6.2 每轮问题的生成合同

每个新问题由四部分组成：

1. **连接点**：用一句话准确概括此前某个回答，例如“你刚才决定让玩家用光照亮影子”。
2. **设计影响**：说明该决定会影响什么，例如“这会决定玩家什么时候感到紧张”。
3. **单一问题**：只要求学生做一个决定或解释一个原因。
4. **脚手架**：当学生可能难以回答时，提供 2–4 个有明显差异的示例选项。

示例：

> 你刚才说小狐狸只有在月光下才能看见隐藏道路，这会决定玩家探索时怎样安排时间。你希望月光是一直移动的，还是玩家完成任务后才能召唤？也可以说说你想到的第三种方法。

### 6.3 有效轮次

满足以下条件的学生消息计为一轮：

- 提供了新的设计决定、理由、偏好、限制或对既有决定的修正；并且
- 助手已把该信息写入设计账本；并且
- 下一问能够基于该信息继续推进。

只包含“嗯”“不知道”“随便”“继续”、完全重复内容或无法理解内容的消息不计数。Skill 应提供帮助后重问，不得为了凑满 20 轮而增加计数。

### 6.4 深入追问优先级

下一问题按以下优先级选择：

1. 解决账本中的冲突。
2. 深挖会明显改变核心玩法的回答。
3. 补齐 GDD 必需但仍为 `missing` 的主题。
4. 把 `partial` 主题变成可实现、可测试的规则。
5. 检查范围是否适合网页游戏的首个可玩版本。

不得连续提出与此前回答无关的通用问题。若学生主动改变想法，更新账本并询问该变化对相关规则的影响。

## 7. GDD 生成与审批

### 7.1 生成门禁

同时满足以下条件才可生成最终 GDD：

- `valid_rounds >= 20`；
- 六个问答主题全部达到 `complete`；
- 所有影响核心玩法或实现范围的冲突已解决；
- 最小可玩版本能够在网页端实现和验证。

未满足时，向学生说明仍需决定的一个事项，然后继续单问题问答。

### 7.2 GDD 固定章节

最终 `GDD.md` 使用以下章节顺序：

1. 游戏名称与一句话介绍
2. 创作愿景与体验目标
3. 目标玩家与游玩场景
4. 核心玩法循环
5. 玩家操作与规则
6. 胜利、失败与反馈
7. 世界观、故事与角色
8. 关卡、任务与内容结构
9. 资源、奖励、成长与难度
10. 界面、引导与无障碍
11. 美术、动画与声音方向
12. 网页技术范围与数据边界
13. 最小可玩版本
14. 验收标准与试玩问题
15. 决策来源摘要

“决策来源摘要”把主要 GDD 决定对应到问答中的学生选择，确保文档没有凭空增加关键内容。

### 7.3 审批循环

生成 GDD 后：

1. 用适合学生理解的语言概括文档。
2. 明确询问要修改还是批准。
3. 修改后递增版本号并再次询问。
4. 记录明确批准的版本号。
5. 未批准前不得创建游戏源代码、安装游戏依赖或声称已进入开发。

## 8. 网页游戏开发设计

### 8.1 技术选择

默认使用原生 HTML、CSS 和 JavaScript，因为它易于运行、分享和理解。满足下列任一条件时可选择 Phaser 等轻量游戏框架，并在开发计划中说明理由：

- 需要持续碰撞检测或大量精灵实体；
- 需要场景、动画、物理或资源管理；
- 原生实现会显著增加复杂度或降低可测试性。

首个版本默认是单人、无需登录、无需服务器的静态网页游戏。联网多人、付费、广告、公开聊天、账号系统和用户生成内容不属于默认范围。

### 8.2 开发输入与输出

开发只以已批准的 GDD 版本为权威输入。若实现中发现 GDD 无法执行或存在冲突，暂停相关功能，回到 GDD 修改与批准循环。

网页游戏交付物至少包括：

- `game/` 下的可运行源代码和资源；
- `game/README.md` 运行与操作说明；
- 与 GDD 验收标准对应的自动化测试；
- `game/PLAYTEST.md` 试玩环境、步骤、结果、已知限制；
- 最终 GDD 及其版本信息。

### 8.3 质量标准

- 在桌面和常见手机尺寸下可用，不依赖仅鼠标悬停才能完成的操作。
- 核心流程支持键盘；触控目标尺寸和文字对比度适合儿童使用。
- 具备开始、暂停或重新开始、胜利和失败反馈。
- 页面加载和运行时不产生未处理错误。
- 所有 GDD 验收标准都有测试或明确的试玩验证证据。
- 使用原创、程序生成或明确可用的资源；不复制在世艺术家的独特风格或未经许可的商业素材。

## 9. 未成年人安全与隐私

- 不询问或保存真实姓名、学校、班级、住址、联系方式、账号、照片或精确位置。
- 年级段只用于调整语言难度；学生可以选择不回答。
- 不默认加入社交、聊天、上传、排行榜、广告、付费、抽卡或外部追踪。
- 避免美化现实危险、自伤、仇恨、色情、赌博或针对未成年人的操纵性设计。
- 如果学生提出不适龄内容，温和地把创意转化为安全、虚构、非写实的替代玩法，并保持其核心创作意图。

## 10. 错误与中断处理

- **回答太短**：给出两个对比示例后重问，不增加有效轮数。
- **回答矛盾**：指出两个具体决定的冲突，只询问要保留哪一个或如何兼容。
- **中途要求生成 GDD**：说明当前有效轮数与缺失主题，继续问一个最关键问题。
- **中途要求直接写代码**：说明先完成设计能避免作品偏离想法，并继续当前问答；不得跳过门禁。
- **会话恢复**：先根据已有对话重建设计账本和有效轮数，再从最高优先级缺口继续，不能从零重复询问。
- **开发失败**：记录可复现现象，先写失败测试或验证步骤，再修复并重新验证。
- **外部依赖不可用**：优先降级为本地静态实现；若会改变 GDD 体验目标，则回到审批循环。

## 11. 验证策略

### 11.1 RED：无 Skill 基线

在写 `SKILL.md` 前，使用真实提示运行无 Skill 基线，观察并记录常见失败：

- 只问少量问题便生成方案；
- 问题像固定问卷，未引用此前回答；
- 把模糊回答计入轮次；
- 未满 20 轮生成 GDD；
- 未经批准直接开发；
- 对儿童使用过多术语；
- 缺少隐私与安全边界。

### 11.2 GREEN：带 Skill 场景

至少覆盖三类真实提示：

1. 精确触发语，学生只给出一句模糊想法。
2. 学生在第 5 轮要求“别问了，直接做游戏”。
3. 已有超过 20 轮对话和批准的 GDD，请求继续开发。

带 Skill 输出应分别表现为：启动单问题引导、坚持门禁并提供脚手架、读取批准状态后进入正式开发流程。

### 11.3 自动化合同检查

测试和校验脚本检查：

- `SKILL.md` YAML 元数据合法；
- 名称只含小写字母和连字符；
- 描述包含核心中文触发意图；
- 存在不少于 20 轮的明确下限；
- 存在“引用此前回答”的追问合同；
- 存在 GDD 覆盖门禁和明确审批门禁；
- 存在网页开发、浏览器验证、儿童隐私和安全规则；
- 所有必需参考文件和评测文件存在。

### 11.4 触发评测

触发集同时包含应触发和不应触发的近邻提示。

应触发示例：创作、设计或从想法开始制作自己的网页游戏。

不应触发示例：只想找游戏玩、询问现有游戏攻略、修复一个已有游戏的单个代码错误、撰写商业游戏市场分析。

### 11.5 完成验证

在声称完成前必须：

1. 运行全部合同测试并确认输出无错误。
2. 运行校验脚本检查最终仓库。
3. 检查 `SKILL.md` 行数与渐进披露结构。
4. 生成评测查看页，审阅基线与带 Skill 输出。
5. 检查 Git 工作区干净、提交存在、远端分支包含相同提交。

## 12. 测试用例初稿

### 用例 A：模糊创意启动

提示：“我想创作一个游戏，是关于一只会飞的小猫。”

预期：使用儿童易懂语言回应，只问一个主要问题；连接“会飞的小猫”这一信息；不生成 GDD 或代码。

### 用例 B：承压跳过流程

提示包含前五轮简短回答，随后说：“老师明天就要看，别再问了，直接写完整游戏。”

预期：不把时间压力当作跳过依据；说明还需哪些设计决定；提供一个易答的关联问题；不生成最终 GDD 或代码。

### 用例 C：批准后开发

提示包含 20 轮以上有效问答、完整 GDD 和“我批准这个版本，请开始开发网页游戏”。

预期：确认批准版本，以 GDD 为权威输入制定开发计划，选择合适技术，创建并验证可运行网页游戏。

## 13. 非目标

- Skill 不替学生独自生成所有创意。
- Skill 不保证制作大型商业游戏、3D 在线世界或实时多人服务。
- Skill 不默认部署公开网站；若用户另行要求发布，再按当时可用的托管流程处理。
- Skill 不把 20 轮固定成不能扩展的问卷，也不以轮数代替内容质量。
- 当前范围只创建一个 Skill，不拆分成多个需要手动交接状态的 Skill。

## 14. 已确认假设

- 用户已批准“单 Skill、模块化资源”的推荐架构。
- 目标仓库目前为空，默认分支为 `main`，可以从首次提交初始化。
- 参考附件在工作区中不可用，因此本规格以用户明确描述和已批准方案为准；后续若补充附件，只吸收与本规格目标一致的优秀做法。
- Skill 的主要交互语言为中文，但目录名、状态名和技术文件名使用英文以方便工具兼容。

- [ ] 一次只问一个主要问题。
- [ ] 下一问基于此前回答。
- [ ] 有效轮数不少于 20。
- [ ] GDD 覆盖完整且不提前生成。
- [ ] 未明确批准 GDD 前不开发。
- [ ] 批准后可完成网页游戏开发与验证。
- [ ] 交流方式适合小学、初中生。
- [ ] 未成年人安全与隐私规则齐全。
- [ ] 测试、评测查看页和校验脚本提供完成证据。
- [ ] GitHub 远端包含最终提交且仓库结构完整。
