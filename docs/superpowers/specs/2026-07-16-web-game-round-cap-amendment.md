# 青少年 Web 游戏创作 Skill：轮次上限与轻量技术栈变更规格

## 1. 变更地位

本规格更新 `youth-game-builder` 的对话轮次和开发技术范围。它在发生冲突时覆盖
`2026-07-16-youth-game-builder-design.md` 中“不少于 20 轮”、可无限延长访谈及
可选重型游戏框架的要求。原规格中的儿童隐私、内容安全、GDD 版本审批、测试先行
和浏览器验证规则继续有效。

## 2. 目标

Skill 最终只交付可在浏览器中运行的 2D Web 游戏。设计访谈应足以形成可开发 GDD，
但不能因为追求完整而无限延长：

- 正常目标是 18–22 个有效回答；
- 第 18 轮是最早正常结束点；
- 第 22 轮是绝对上限，之后不得继续提问；
- 学习者发送“停止”时立即结束访谈，不再追加问题；
- 达到上限或收到“停止”后，未决内容采用安全、常见、轻量的默认设计补齐；
- “停止”只结束设计访谈，不代表批准 GDD，也不授权开发。

## 3. 状态模型

在现有设计账本中增加：

```text
target_rounds: 18..22
hard_round_cap: 22
stop_requested: true | false
completion_reason: coverage-complete | round-cap | stop
defaulted_decisions: [{field, value, reason, source: assistant-default}]
```

`valid_rounds` 的原定义保持不变：只有学习者新增或修改设计决定、理由、偏好或约束
时才增加；“停止”本身不计为有效回答。

## 4. 自适应访谈算法

### 4.1 正常路径

1. `valid_rounds < 18` 时，一次只问一个关联问题。
2. `18 <= valid_rounds < 22` 时：
   - 六个主题完整、核心冲突已解决且 Web MVP 可实施，则停止提问并生成 GDD；
   - 否则只询问当前最关键的缺口。
3. `valid_rounds == 22` 时，无论覆盖是否完全，都不得再提问。采用默认设计补齐缺口、
   选择最安全简单的方案解决剩余冲突，然后生成 GDD。
4. 正常流程不得产生第 23 个有效问答请求。

目标不是机械追求 22 轮，而是在 18 轮后尽早完成；只有关键缺口才延长。

### 4.2 “停止”路径

把去除首尾空白后等于“停止”的学习者消息视为明确停止指令。也可接受语义完全明确
的“停止提问”或“结束问答”，但不得把“暂停游戏”“角色停止移动”等游戏设计内容
误判为停止访谈。

收到停止指令后：

1. 设置 `stop_requested: true` 和 `completion_reason: stop`；
2. 不再提出任何设计问题；
3. 保留全部学习者已确认决定；
4. 用第 5 节默认方案补齐缺失字段并解决冲突；
5. 直接生成完整 GDD，状态仍为 `待批准`、`gdd_approved: false`；
6. 明确列出哪些内容来自学习者，哪些是系统默认；
7. 等待学习者明确批准当前 GDD 版本后才开发。

## 5. 主流默认设计

默认值只用于学习者未决定的部分，不得覆盖已确认决定：

| 未决内容 | 默认方案 |
|---|---|
| 游戏类型 | 单人 2D 轻量闯关或收集游戏 |
| 单局时长 | 3–5 分钟 |
| 核心循环 | 移动或点击 → 收集/避障 → 即时反馈 → 到达目标 |
| 输入 | 方向键/WASD + 空格；手机提供等价触控按钮 |
| 胜利 | 达成关卡目标或收集指定数量 |
| 失败 | 资源归零或时间结束；从当前关卡快速重试 |
| 难度 | 3 个短关卡逐步增加速度、数量或组合 |
| 视觉 | 高对比度原创几何图形、少量动画 |
| 声音 | 可关闭的简短程序生成或许可明确音效 |
| 数据 | 无登录、无服务器、无上传、无追踪；仅在必要时使用本地设置 |
| 无障碍 | 可见焦点、键盘可完成、触控等价、支持减少动态效果 |
| 验收 | 能开始、暂停、胜利、失败、重试；键盘/触控可用；零控制台错误 |

GDD 的决策来源表新增来源类型 `学习者决定 | 系统默认`。系统默认必须附简短理由，
不能伪装成学习者回答。

## 6. Web-only 技术合同

### 6.1 最终形态

最终产物必须是静态、离线可运行的 2D Web 游戏，入口为 `game/index.html`。不得生成
桌面安装包、原生移动应用、Unity/Unreal/Godot 工程或需要云端服务才能游玩的版本。

### 6.2 默认技术栈

- 原生 HTML5、CSS 和 JavaScript；
- 使用 DOM、Canvas 2D 或二者组合；
- 使用普通脚本或已本地合并的单一脚本，避免运行时 `import`、`fetch` 或跨域请求；
- 所有资源随 `game/` 本地交付，不依赖 CDN；
- 玩家无需安装 Node.js、Python、包管理器或数据库；
- `index.html` 应能在文件管理器中直接双击离线运行；
- 开发与自动测试可使用本地 HTTP 服务，但这不是玩家运行前提。

### 6.3 禁止的默认依赖

默认不使用：

- WebGL、3D 引擎、WASM、GPU 计算或高开销粒子/后处理；
- Phaser 等游戏框架；
- React/Vue 等应用框架；
- npm 构建链、打包器或运行时依赖安装；
- 后端、数据库、登录、云函数、联网多人、遥测或外部 API；
- 大型视频、高清纹理包、远程字体或总量超过 10 MB 的首版资源。

只有学习者明确提出且新的 GDD 说明必要性、仍能满足兼容和资源预算并重新获批时，
才可使用一个轻量本地依赖；仍不得破坏离线运行。

### 6.4 兼容目标

游戏应在多数 Windows 10/11 与近年 macOS 电脑的主流浏览器中运行：

- Windows：当前稳定版 Chrome、Edge 或 Firefox；
- macOS：当前稳定版 Safari、Chrome 或 Firefox；
- 不要求独立显卡；
- 在桌面 1280×720 和手机 390×844 视口完成核心流程；
- 键盘和触控提供等价操作；
- 首版资源总量不超过 10 MB；
- 避免持续高负载动画，普通设备目标 60 FPS，低性能设备不得低于可玩的 30 FPS。

## 7. 交付与验证

现有 `game/README.md`、自动测试、`game/PLAYTEST.md` 和审批门槛保持不变，并增加：

- README 第一运行方式是“双击 `index.html`”；
- 测试确认没有运行时网络请求、CDN 或服务端依赖；
- 在 Windows 风格与 macOS 风格的路径/浏览器条件下检查相对资源；
- PLAYTEST 记录离线启动、资源总量、桌面浏览器兼容和性能结果；
- 若游戏不能直接离线打开，不得标记完成。

## 8. 评估场景

至少覆盖：

1. 第 18 轮时信息完整：停止提问并生成待批准 GDD。
2. 第 18 轮时仍有关键缺口：继续一个问题，但不超过第 22 轮。
3. 第 22 轮仍有缺口：不再提问，默认补齐并生成 GDD。
4. 第 7 轮收到“停止”：不再提问，默认补齐，生成待批准 GDD，不开发。
5. 已批准 GDD 开发：产物是可双击运行、无复杂依赖的 2D Web 游戏。

## 9. 验收标准

- [ ] 运行时 Skill、README 和引用合同均明确最终只生成 Web 游戏。
- [ ] 正常访谈保持在 18–22 个有效回答，绝不请求第 23 轮。
- [ ] “停止”立即终止提问并触发默认补齐。
- [ ] 默认补齐内容在 GDD 中可追溯，且不冒充学习者决定。
- [ ] “停止”不被视为 GDD 批准。
- [ ] 默认技术栈是原生 HTML/CSS/JavaScript，玩家无需安装依赖。
- [ ] `index.html` 可以直接双击离线运行。
- [ ] 不默认使用高性能或复杂技术栈。
- [ ] 支持多数 Windows/Mac 主流浏览器。
- [ ] 原有隐私、安全、明确批准、测试与验证要求仍有效。
